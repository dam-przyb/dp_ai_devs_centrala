# Example Document: Introduction to Retrieval-Augmented Generation

## What is RAG?

Retrieval-Augmented Generation (RAG) is a technique that enhances large language model (LLM) responses by providing relevant external knowledge at query time. Instead of relying solely on the model's training data, RAG retrieves documents from a knowledge base and includes them in the model's context.

The core idea is simple: when a user asks a question, the system first searches a document store to find relevant passages, then passes those passages alongside the question to the LLM. This allows the model to ground its answers in up-to-date, verifiable information.

## Why RAG Matters

Traditional LLMs have several limitations:
- Knowledge cutoff dates mean they cannot answer questions about recent events.
- They can hallucinate plausible but incorrect facts.
- They cannot access proprietary or domain-specific information.

RAG addresses all three of these issues by separating the knowledge store from the model itself. This makes it easy to update the knowledge base without retraining the model.

## Chunking Strategies

Before documents can be indexed for retrieval, they must be split into manageable pieces called chunks. The choice of chunking strategy significantly affects retrieval quality.

### Fixed-Size Character Chunking

The simplest approach splits text every N characters with an optional overlap. A typical configuration uses 1000-character chunks with 200-character overlap. This guarantees uniform chunk sizes but may split sentences or paragraphs mid-thought.

### Separator-Based Chunking

A more sophisticated approach uses a hierarchy of separators: first splitting on major section headings (`## `), then paragraphs (`\n\n`), then sentences. This preserves semantic boundaries and keeps related content together. Section headings extracted during splitting can be stored as metadata to improve retrieval.

### Context-Enriched Chunking

Each chunk receives a short (1-2 sentence) context generated by the LLM that situates the chunk within the overall document. For example, a chunk about "database indexing" from a chapter on "performance optimization" would receive context like: "This passage describes indexing strategies within the broader chapter on database performance." This context is prepended to the chunk before embedding, improving retrieval recall.

### Topic-Based Chunking

Rather than splitting mechanically, a single LLM call is made to decompose the entire document into logically coherent topics. Each resulting chunk represents one coherent idea. The original text is preserved verbatim — the LLM only identifies boundaries, not rewrite content.

## Embedding Models

Once chunks are created, they are converted to dense vector representations using an embedding model. The `text-embedding-3-small` model from OpenAI produces 1536-dimensional vectors and offers an excellent balance of cost and quality for most RAG applications.

Cosine similarity is the standard metric for comparing embeddings. Two vectors pointing in nearly the same direction (similarity ≥ 0.60) indicate semantically similar content. Moderate similarity (0.35–0.60) suggests related but not identical topics. Low similarity (below 0.35) indicates largely unrelated content.

## Hybrid Retrieval

Hybrid RAG combines two complementary search methods:

### BM25 / Full-Text Search

BM25 is a classic term-frequency ranking algorithm used in engines like Elasticsearch and SQLite FTS5. It excels at finding exact keyword matches and is language-agnostic. It is fast, requires no GPU, and degrades gracefully.

### Vector Search

Dense retrieval uses embedding similarity to find semantically related content even when no exact keywords match. This is especially useful for paraphrased queries or multilingual content.

### Reciprocal Rank Fusion (RRF)

RRF is a simple but effective technique to merge ranked lists from multiple retrieval methods. For each document, its score is the sum of `1 / (k + rank)` across all lists where it appears. A typical value of `k = 60` prevents highly-ranked documents in one list from completely dominating the fusion score.

## SQLite as a Vector Store

SQLite combined with the `sqlite-vec` extension provides a lightweight but capable vector store that requires no external services. All data — documents, FTS index, and embedding vectors — lives in a single `.db` file. This makes it ideal for local development and small-to-medium scale deployments.

## Summary

RAG is now a foundational pattern in modern AI applications. Understanding the tradeoffs between chunking strategies, embedding models, and retrieval methods allows engineers to build systems that are accurate, fast, and maintainable without requiring expensive retraining cycles.
