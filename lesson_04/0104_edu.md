# Lesson 04 — SPK Transport Declaration Agent
## Educational Notes: Agent Design, Tool Use, Vision, and Django Integration

---

## 1. What We Built

A Django-integrated LLM agent that autonomously:

1. Reads a local documentation index (`index.md`) and discovers all companion files it references
2. Fetches those files from a remote URL — routing **image files through a vision model** to extract their text content
3. Assembles the full documentation set and injects it into an LLM agent's system prompt as context
4. Runs a **tool-calling agent loop** that reasons about the task, fills an exact template, and submits to a verification endpoint
5. **Self-corrects on rejection** — reads the hub's error message, adjusts the declaration, and retries (up to 3 attempts)
6. Logs every action to a persistent `log.json` file and surfaces everything in a Django/HTMX frontend

The agent found the flag `{FLG:WISDOM}` on the third submission attempt.

---

## 2. Agent Architecture: The Core Pattern

This lesson uses the **Supervisor (single-tool) agent** pattern — the simplest form of tool-calling agent. Unlike the multi-tool findhim agent from lesson 02, this agent has exactly **one tool**: `submit_declaration`. All the reasoning, document parsing, and template filling happens purely in the LLM's context window using the injected documentation.

```
┌─────────────────────────────────────────────────────────────────┐
│                         Agent Loop                              │
│                                                                 │
│   System Prompt                                                 │
│   ┌────────────────────────────────────────────────────────┐   │
│   │  Task parameters (sender, route, weight, budget)       │   │
│   │  Full SPK documentation (index.md + all attachments)  │   │
│   │  Step-by-step reasoning instructions                   │   │
│   └────────────────────────────────────────────────────────┘   │
│                            │                                    │
│                            ▼                                    │
│   HumanMessage: "Fill the form and submit"                      │
│                            │                                    │
│                            ▼                                    │
│   ┌─────────────────────────────────┐                          │
│   │   LLM Iteration N               │◄──── ToolMessage(result) │
│   │   Reasoning → tool_call?        │                          │
│   └────────────────┬────────────────┘                          │
│                    │  tool_calls = [submit_declaration(...)]   │
│                    ▼                                            │
│   ┌─────────────────────────────────┐                          │
│   │   submit_declaration(text)      │                          │
│   │   POST /verify → HTTP response  │──── result back to LLM  │
│   └─────────────────────────────────┘                          │
│                                                                 │
│   Exit when: flag found | no tool call | max attempts reached   │
└─────────────────────────────────────────────────────────────────┘
```

The key insight: **the LLM is not just calling tools — it is also the parser, the reasoning engine, and the error interpreter**. When the hub returned `"The shipment will not fit on the train."`, the LLM read that and correctly concluded it should add an extra-wagon declaration field in the next submission.

## 3. Pre-run Document Fetching (Before the Agent Starts)

The SPK documentation was split across multiple files linked with a custom directive:

```markdown
[include file="zalacznik-E.md"]
[include file="trasy-wylaczone.png"]
```

**Key design decision**: All file fetching was done *before* the agent loop starts, in pure Python. This avoids giving the agent tools for HTTP fetching, which would make the loop more complex and harder to observe. The LLM only sees the assembled result.

### 3.1 Parsing `[include]` Directives

```python
def _extract_include_files(text: str) -> list[str]:
    # Regex scans for all [include file="..."] patterns in the markdown
    seen: set[str] = set()
    result: list[str] = []
    for match in re.finditer(r'\[include file="([^"]+)"\]', text):
        fname = match.group(1)
        if fname not in seen:
            seen.add(fname)
            result.append(fname)  # preserve document order, deduplicate
    return result
```

10 companion files were discovered: `trasy-wylaczone.png`, `dodatkowe-wagony.md`, `zalacznik-A.md` through `zalacznik-H.md`.

### 3.2 Fetching Text Documents

Straight `httpx.get()` with a 20s timeout. Errors are caught and returned as descriptive strings rather than raising — the agent can see the error and adapt.

```python
def _fetch_text_doc(filename: str) -> str:
    url = f"{_DOC_BASE_URL}/{filename}"  # https://hub.ag3nts.org/dane/doc/{filename}
    try:
        resp = httpx.get(url, timeout=20)
        resp.raise_for_status()
        return resp.text
    except Exception as exc:
        return f"[ERROR fetching {filename} from {url}: {exc}]"
```

### 3.3 Fetching and Analysing Image Documents (Vision)

The blocked routes table was only available as a PNG image. The solution: fetch the image bytes, base64-encode them, then send them to `gpt-4o` as an **inline data URL** within a multimodal message.

```python
def _analyze_image_doc(filename: str) -> str:
    # 1. Fetch the image as raw bytes
    resp = httpx.get(url, timeout=20)
    image_bytes = resp.content

    # 2. Base64-encode so we can embed it in the message
    b64 = base64.b64encode(image_bytes).decode()
    media_type = "image/png"

    # 3. Send to gpt-4o as a multimodal HumanMessage
    vision_llm = _llm(model="openai/gpt-4o")
    message = HumanMessage(
        content=[
            {
                "type": "text",
                "text": "Extract ALL visible content: text, tables, route codes...",
            },
            {
                "type": "image_url",
                "image_url": {"url": f"data:{media_type};base64,{b64}"},
            },
        ]
    )
    response = vision_llm.invoke([message])
    return f"[Content extracted from image: {filename}]\n\n{response.content}"
```

**Why inline base64 rather than passing the URL directly?** Two reasons:
- The hub URL might not be publicly image-accessible to OpenRouter's servers
- Inline embedding works regardless of URL accessibility or authentication

The extracted image content (route codes including the `X-01: Gdańsk–Żarnowiec` entry) was then inserted into the agent's context just like a text document.

## 4. The Agent Tool: `submit_declaration`

Defined as a **LangChain `@tool`** — a decorated function that the LLM can call by name with typed arguments. The tool is defined *inside* `run_sendit_agent()` so it has closure access to shared mutable state (`action_log`, `submissions`, `flag_found`) without needing global variables or a class.

```python
@tool
def submit_declaration(declaration: str) -> str:
    """
    Submit the filled SPK transport declaration to the verification hub.
    ...
    """
    nonlocal flag_found, final_declaration  # write back to outer scope

    payload = {
        "apikey": _api_key(),
        "task": "sendit",
        "answer": {"declaration": declaration},
    }

    resp = httpx.post(f"{_HUB_BASE}/verify", json=payload, timeout=30)
    body = resp.text

    # Log every attempt immediately — before returning to LLM
    _append_log(action_log, {...})
    _save_log(action_log)  # flush to disk

    # Search the response for the flag pattern
    flag_match = re.search(r"\{FLG:[^}]+\}", body)
    if flag_match:
        flag_found = flag_match.group(0)

    # Return the raw hub response — the LLM reads and interprets this
    return f"[Attempt {attempt_num}/{MAX_SUBMISSION_ATTEMPTS}] HTTP {resp.status_code}: {body}"
```

**Critical design choice**: The tool returns the **raw hub error message** as the tool result string. This becomes a `ToolMessage` appended to the conversation, so the LLM reads the hub's explanation and decides how to fix the declaration on the next iteration.

From `log.json`, you can see this self-correction in action:
- **Attempt 1** → `HTTP 400: {"code": -940, "message": "Missing or invalid date (expected format: YYYY-MM-DD)."}` — LLM had put `[2023-10-05]` with brackets; it removed them
- **Attempt 2** → `HTTP 400: {"code": -760, "message": "The shipment will not fit on the train."}` — LLM correctly deduced it needed to declare extra wagons for 2800 kg
- **Attempt 3** → `HTTP 200: {"code": 0, "message": "{FLG:WISDOM}"}` ✓

## 5. The Agent Loop

The loop follows the standard LangChain tool-calling pattern: invoke LLM → dispatch tool calls → append results → repeat.

```python
# Bind the tool so the LLM knows it can call it
llm = _llm(model=SENDIT_MODEL).bind_tools([submit_declaration])
tool_map = {"submit_declaration": submit_declaration}

messages = [
    SystemMessage(content=system_prompt),   # full docs + task params here
    HumanMessage(content="Fill the form and submit..."),
]

while iteration < MAX_AGENT_ITERATIONS:
    iteration += 1
    response = llm.invoke(messages)
    messages.append(response)              # append AI response to history

    if not response.tool_calls:
        break                              # LLM finished without calling tools

    if submission_state["attempts"] >= MAX_SUBMISSION_ATTEMPTS and not flag_found:
        break                              # all retries used

    for tc in response.tool_calls:
        tool_fn = tool_map.get(tc["name"])
        tool_result = tool_fn.invoke(tc["args"])
        messages.append(
            ToolMessage(content=str(tool_result), tool_call_id=tc["id"])
        )

    if flag_found:
        break                              # success — stop immediately
```

The `tool_call_id` on `ToolMessage` is critical — it links the tool result back to the specific call that requested it. Omitting it causes most models to reject the message structure.

### Why `bind_tools` instead of function calling directly?

`bind_tools([...])` generates the OpenAI `tools` schema from each function's signature and docstring automatically. The LLM sees the function name, argument names, types, and the docstring description — all of which help it call the tool correctly. A good docstring is not just documentation; for tool-calling agents it is **part of the prompt**.

## 6. System Prompt: Reasoning Context

The system prompt had two parts:

**Part 1 — explicit reasoning guidance** (hardcoded instructions about category, route logic, fee calculation):
```
2. Determine the correct CATEGORY:
   - Reactor fuel cassettes ("kasety z paliwem do reaktora" / "ogniwa paliwowe")
     fall under Category A — Strategic.
   - Category A is funded by the System (0 PP base cost).
   - Only Category A and B may use normally blocked routes.
```

**Part 2 — the full documentation corpus** (~45,000 characters of text injected directly):
```python
full_docs = "\n".join([
    f"=== MAIN DOCUMENTATION: index.md ===\n\n{index_content}",
    f"=== COMPANION FILE: zalacznik-E.md ===\n\n{fetched_docs['zalacznik-E.md']}",
    # ... all 10 fetched files
])
system_prompt = f"...instructions...\n\nCOMPLETE SPK DOCUMENTATION:\n{full_docs}"
```

This approach — **context stuffing** — works well when the entire documentation fits within the model's context window (gpt-4o: 128k tokens). For larger corpora you would need RAG (retrieval-augmented generation) instead.

## 7. Persistent Logging Pattern

Every significant action writes to `log.json` **immediately after it happens**, not at the end. This means:
- If the agent crashes mid-run, you get a partial log showing exactly where it failed
- Each entry carries a UTC timestamp for performance analysis

```python
def _append_log(log: list[dict], entry: dict) -> None:
    log.append({**entry, "ts": datetime.now(timezone.utc).isoformat()})

def _save_log(log: list[dict]) -> None:
    _LOG_PATH.write_text(
        json.dumps(log, ensure_ascii=False, indent=2),
        encoding="utf-8",
    )
```

The log is also returned to the view in the result dict, making it available directly in the template without re-reading the file.

## 8. Django Integration

### View (thin layer)
```python
# views/sendit.py
@require_POST
def sendit_api(request: HttpRequest) -> HttpResponse:
    try:
        data = run_sendit_agent()
        return render(request, "lesson_04/partials/sendit_result.html", {"data": data})
    except Exception as exc:
        return render(request, "lesson_04/partials/sendit_result.html", {"error": str(exc)})
```

The view does nothing except call the service and route the result. All logic lives in `sendit_service.py`.

### HTMX Pattern
The frontend uses a single HTMX form to trigger the long-running agent via POST and swap the result in-place when it finishes:

```html
<form hx-post="{% url 'l04_sendit_api' %}"
      hx-target="#sendit-result"
      hx-swap="innerHTML"
      hx-indicator="#sendit-spinner">
  ...
  <span id="sendit-spinner" class="htmx-indicator animate-pulse">
    Fetching docs, running agent… (up to 90 seconds)
  </span>
</form>
<div id="sendit-result"></div>
```

`hx-indicator` points to a spinner element that shows automatically while the request is in progress (HTMX adds the `htmx-request` class during the fetch) and hides when the response arrives. No JavaScript needed.

### Template Filter Gotcha
Django does **not** have a `|tojson` template filter (that is a Jinja2/Flask convention). For safely embedding Python strings in inline `onclick` JavaScript, use `|escapejs` with explicit surrounding single quotes:

```html
{# WRONG — tojson does not exist in Django #}
<button onclick="navigator.clipboard.writeText({{ text|tojson }})">Copy</button>

{# CORRECT — escapejs escapes backslashes, quotes, newlines etc. #}
<button onclick="navigator.clipboard.writeText('{{ text|escapejs }}')">Copy</button>
```

## 9. Key Concepts Summary

| Concept | Where used | Key takeaway |
|---|---|---|
| Tool-calling agent | `run_sendit_agent()` loop | LLM calls `submit_declaration`; result feeds back as `ToolMessage` |
| Multimodal / Vision | `_analyze_image_doc()` | Base64-encode image → inline `data:` URL → gpt-4o extracts text |
| Context stuffing | System prompt assembly | Full docs injected as text; works up to ~128k token limit |
| Closure-based tool | `@tool` inside function | Avoids globals; tool shares `action_log`, `flag_found` with outer scope |
| `bind_tools` | LLM setup | Auto-generates OpenAI function schema from signature + docstring |
| `ToolMessage` with `tool_call_id` | Loop dispatch | Required to correctly link results back to specific tool calls |
| Incremental log flush | `_save_log()` every step | Crash-safe audit trail |
| HTMX `hx-indicator` | `sendit.html` | Zero-JS loading spinner for long-running agent requests |
| `|escapejs` not `|tojson` | Template | Django-specific — use `escapejs` for inline JS string embedding |
