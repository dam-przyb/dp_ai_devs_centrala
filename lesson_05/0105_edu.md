# Lesson 05 — Orchestration: Railway X-01 Quest

## What was built

A fully autonomous LLM agent that activates railway route X-01 by exploring a
self-documenting API, handling server-side chaos (503 errors and aggressive rate
limits), and streaming its entire reasoning process to the browser in real time.

---

## 1. The Problem

The task provides a single entry point: `POST https://hub.ag3nts.org/verify` with
`{"action": "help"}`.  Everything else — action names, parameter names, required
sequence — must be discovered at runtime by reading the `help` response.
Two additional constraints make it non-trivial:

| Constraint | What it means in practice |
|---|---|
| Deliberate 503 responses | Not a real error — must be retried automatically |
| Strict rate limits | Quota headers must be read after every call; violating them causes long blocks |

---

## 2. High-Level Architecture

```
Django view (POST /05/railway/api/)
  │
  ├─► run_railway_agent()         returns task_id immediately
  │        │
  │        └─► daemon thread → LangGraph graph
  │
  └─► returns railway_started.html (contains EventSource JS)

Browser ──────────────────────────────────────► GET /05/railway/stream/<task_id>/
                                                    │   (SSE — text/event-stream)
                                                    │   polls _task_store every 0.3 s
                                                    └── pushes JSON events to browser
```

The HTTP request that starts the agent returns **immediately** — there is no
blocking wait in the Django view.  All work happens in a background daemon thread.
The browser connects to a separate SSE endpoint to watch the agent work in real time.

## 3. LangGraph Agent — Supervisor + ReAct Worker

### 3.1 Graph topology

```
START ──► railway_worker_node ──► supervisor_node ──► END
                 ▲                       │
                 └───────────────────────┘
                   (loop: no flag found yet
                    AND round cap not reached)
```

Two nodes collaborate:
- **`railway_worker_node`** — the "doer": calls the LLM and executes tool calls
- **`supervisor_node`** — the "checker": scans results for the flag and decides
  whether to keep looping or terminate

### 3.2 State schema

```python
class RailwayState(TypedDict):
    messages:          list        # full conversation — System + Human + AI + Tool messages
    flag:              str | None  # captured {FLG:...} string; None until found
    supervisor_rounds: int         # incremented by supervisor on each pass
    task_id:           str         # key into _task_store for live logging
```

Keeping `task_id` in the graph state lets every node log against the
correct in-memory store without global variables.

### 3.3 Worker node — manual ReAct loop

Rather than using `langgraph.prebuilt.create_react_agent` (which hides the
tool-execution loop), the worker implements the loop explicitly.
This makes it easy to intercept and log every step:

```python
def railway_worker_node(state: RailwayState) -> dict:
    # Bind task_id to thread-local so the @tool function can log without
    # receiving task_id as an explicit LLM-visible argument.
    _thread_local.task_id = state["task_id"]

    llm_with_tools = llm.bind_tools([call_railway_api])

    for iteration in range(1, MAX_WORKER_ITERATIONS + 1):
        response = llm_with_tools.invoke(messages)
        messages.append(response)

        if not response.tool_calls:
            # LLM produced a final answer — exit the loop
            break

        for tc in response.tool_calls:
            result_str = tools_by_name[tc["name"]].invoke(tc["args"])
            messages.append(
                ToolMessage(content=result_str, tool_call_id=tc["id"])
            )

    return {"messages": messages}
```

The pattern:
1. Invoke LLM with the full message history
2. If the response contains tool calls → execute each tool → append `ToolMessage` → repeat
3. If the response contains no tool calls → the LLM has produced its final answer → break

### 3.4 Supervisor node

```python
def supervisor_node(state: RailwayState) -> dict:
    rounds = state["supervisor_rounds"] + 1

    # Scan all messages (newest first) for {FLG:...}
    for msg in reversed(state["messages"]):
        match = _FLAG_RE.search(getattr(msg, "content", "") or "")
        if match:
            return {"flag": match.group(0), "supervisor_rounds": rounds}

    if rounds >= MAX_SUPERVISOR_ROUNDS:
        return {"supervisor_rounds": rounds}   # terminate with no flag

    # No flag yet — nudge the worker and keep looping
    nudge = HumanMessage(content=f"Round {rounds}: continue following the API docs...")
    return {"messages": state["messages"] + [nudge], "supervisor_rounds": rounds}
```

The conditional edge decides routing:

```python
def _should_continue(state: RailwayState) -> str:
    if state.get("flag"):
        return END
    if state["supervisor_rounds"] >= MAX_SUPERVISOR_ROUNDS:
        return END
    return "railway_worker"
```

### 3.5 System prompt strategy

```
You are a precise API automation agent. Your mission: activate railway route X-01.

STRICT RULES
1. Begin by calling call_railway_api(action="help").
2. Read the documentation carefully.
3. Use ONLY the exact action names and parameter names that help returned.
4. Never guess or invent action names or parameter names.
5. When any response contains {FLG:...}, report it as your FINAL ANSWER.
```

The prompt is deliberately constraining — it prevents the LLM from "hallucinating"
action names, which would burn rate-limit quota on invalid calls.

## 4. The LangChain Tool

```python
@tool
def call_railway_api(action: str, params: dict | None = None) -> str:
    """
    Call the railway hub API at hub.ag3nts.org.

    IMPORTANT: Always start with action='help' to receive the official API
    documentation before calling any other action.
    """
    answer: dict = {"action": action, **(params or {})}
    payload = {"apikey": api_key, "task": "railway", "answer": answer}
    _, body, _ = _call_hub(payload)
    return json.dumps(body)
```

Key design choices:
- **`params: dict | None`** — a single flexible dict rather than typed individual
  parameters, because the parameter names are discovered at runtime from `help`.
  If the tool had named parameters like `route: str`, the LLM would be unable to
  pass parameters it learns about later.
- **Docstring** tells the LLM *what to do first* (`help`) and *what not to do*
  (guess names).  LLMs read tool docstrings to decide how and when to invoke them.
- **Thread-local task_id** — the tool cannot accept `task_id` as a parameter
  (the LLM would try to fill it in as an argument).  Instead, the worker node
  writes `_thread_local.task_id` before invoking the LLM, and the tool reads it:

```python
_thread_local = threading.local()

def _current_task_id() -> str:
    return getattr(_thread_local, "task_id", "unknown")
```

## 5. Resilient HTTP Layer

### 5.1 503 exponential back-off (`_call_hub`)

```python
backoff = INITIAL_BACKOFF_S   # 2 s

for attempt in range(1, MAX_503_RETRIES + 1):   # up to 8 attempts
    resp = httpx.post(HUB_URL, json=payload)

    if resp.status_code == 503:
        retry_after = resp.headers.get("retry-after")
        wait_s = max(int(retry_after), backoff) if retry_after else backoff
        time.sleep(wait_s)
        backoff = min(backoff * 2, 120)   # double, cap at 120 s
        continue

    _handle_rate_limit(resp.headers)
    return resp.status_code, resp.json(), resp.headers
```

Why exponential back-off?  If the server is genuinely overloaded each attempt
should wait longer than the last.  The cap at 120 s prevents indefinite hangs.

### 5.2 Rate-limit header handling (`_handle_rate_limit`)

The API sends non-standard headers that require interpretation:

| Header | Meaning | Gotcha |
|---|---|---|
| `retry-after` | Standard HTTP — *seconds to wait* | Always reliable |
| `x-ratelimit-remaining` | Calls left in quota window | `"0"` means quota exhausted |
| `x-ratelimit-reset-after` | When quota resets | **Can be an absolute Unix timestamp, not a duration!** |

Resolution priority:
1. `retry-after` (standard, always in seconds)
2. `x-ratelimit-reset-after` — checked with a heuristic:
   - value > 3 600 → treat as Unix timestamp → `wait = value - time.time()`
   - value ≤ 3 600 → treat as duration in seconds
3. Fallback: 5 s if quota is zero but no timing header exists

Hard cap applied at the end regardless:

```python
wait_s = min(wait_s, MAX_RATE_LIMIT_WAIT_S)   # never sleep more than 120 s
```

Without this cap, the `x-ratelimit-reset-after: 1775333535` (a Unix timestamp
representing April 2027) would have caused a **56-year sleep**.

## 6. Live Logging Architecture

### 6.1 In-memory task store

```python
_task_store: dict[str, dict] = {}
# { task_id: { "log": [...], "done": bool, "flag": str|None, "error": str|None } }
```

The background agent thread **appends** to `store["log"]`.
The SSE view **reads** from it.
CPython's GIL makes `list.append` and dict writes effectively atomic, so no
locking is needed for this single-writer/single-reader pattern.

### 6.2 SSE stream view

```python
def railway_stream(request, task_id):
    def _generate():
        last_index = 0
        while True:
            state = get_task_state(task_id)
            # Push any new log entries since last poll
            while last_index < len(state["log"]):
                yield "data: " + json.dumps(state["log"][last_index]) + "\n\n"
                last_index += 1
            if state["done"]:
                yield "data: " + json.dumps({"type": "done", ...}) + "\n\n"
                return
            time.sleep(SSE_POLL_INTERVAL_S)   # 0.3 s

    return StreamingHttpResponse(_generate(), content_type="text/event-stream")
```

`StreamingHttpResponse` with `content_type="text/event-stream"` keeps the HTTP
connection open.  The `X-Accel-Buffering: no` header disables nginx/proxy
buffering so events arrive immediately.

### 6.3 Browser-side EventSource

```javascript
const source = new EventSource("/05/railway/stream/<task_id>/");

source.onmessage = function(evt) {
    const data = JSON.parse(evt.data);

    if (data.type === "done") {
        source.close();
        // show flag banner or error
        return;
    }
    appendEntry(data);   // render into the live timeline
};
```

Each log entry has a `type` field that determines colour and icon in the UI:

| type | Colour | Meaning |
|---|---|---|
| `agent_step` | slate | LLM decision or worker lifecycle |
| `api_call` | gray | Request about to be sent |
| `api_response` | blue | Response received |
| `retry_503` | amber | 503 hit, waiting to retry |
| `rate_limit` | orange | Quota exhausted, sleeping |
| `flag` | green | `{FLG:...}` captured |
| `error` | red | Any failure |
| `done` | green | Terminal event — stream closes |

## 7. Background Thread + Artifact Saving

```python
def run_railway_agent(model=None) -> str:
    task_id = str(uuid.uuid4())
    _task_store[task_id] = {"log": [], "done": False, "flag": None, "error": None}

    thread = threading.Thread(
        target=_run_agent_thread,
        args=(task_id, model),
        daemon=True,           # thread dies if the Django process exits
    )
    thread.start()
    return task_id             # returned to the view immediately
```

On completion (success or failure), `_save_artifacts` writes to disk:

```python
def _save_artifacts(task_id):
    log_path = _ANSWER_DIR / "log.json"
    with log_path.open("w") as fh:
        json.dump(state["log"], fh, indent=2)

    if flag := state.get("flag"):
        answer_path = _ANSWER_DIR / "answer.json"
        with answer_path.open("w") as fh:
            json.dump({"task": TASK_NAME, "flag": flag}, fh, indent=2)
```

## 8. URL and View Layer

Three endpoints per the established lesson pattern:

| Method | URL | Purpose |
|---|---|---|
| GET | `/05/railway/` | Render workspace partial |
| POST | `/05/railway/api/` | Start agent thread, return live-log partial |
| GET | `/05/railway/stream/<task_id>/` | SSE stream — push log events |

The POST view returns in milliseconds because the agent runs in a background
thread.  The browser immediately connects to the SSE endpoint and begins
receiving events.
